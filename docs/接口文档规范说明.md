# 接口文档规范说明

下面是一份针对 **任务管理模块** 的接口规范文档，涵盖「创建任务」与「查询任务状态」两个核心接口。文档遵循 RESTful 设计风格，字段命名统一采用 `snake_case`，便于前后端协作对齐 👇

---

## 📘 通用约定

在进入具体接口之前，先明确一些通用规范，避免后续歧义。

### 基础信息

| **项目**     | **说明**                          |
| ------------ | --------------------------------- |
| **协议**     | HTTPS                             |
| **请求格式** | `application/json`                |
| **响应格式** | `application/json; charset=utf-8` |
| **字符编码** | UTF-8                             |
| **命名规范** | 字段使用 `snake_case`             |

### 通用响应结构

所有接口均采用统一的外层包装结构：

```json
{
  "code": 0,
  "message": "success",
  "data": { }
}
```

| **字段**  | **类型** | **说明**                                  |
| --------- | -------- | ----------------------------------------- |
| `code`    | int      | 业务状态码，`0` 表示成功，非 `0` 表示失败 |
| `message` | string   | 结果描述信息                              |
| `data`    | object   | 业务数据主体，不同接口结构不同            |

---

## 🚀 1. 创建任务接口

用于提交一个新的训练 / 计算任务，服务端异步执行，仅返回任务唯一标识 `job_id`，后续通过查询接口轮询状态。

### 接口信息

| **项目**     | **说明**             |
| ------------ | -------------------- |
| **接口路径** | `/api/v1/train_jobs` |
| **请求方法** | `POST`               |
| **鉴权方式** | Bearer Token         |

### 请求参数

请求体字段由业务场景决定（此处示意），核心是任务创建所需的配置信息

### 响应示例

创建成功后，服务端仅返回 `job_id`，不包含其他业务字段。

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "job_id": "j_20260424_abc123"
  }
}
```

### 响应字段说明

| **字段** | **类型** | **是否必返** | **说明**                                 |
| -------- | -------- | ------------ | ---------------------------------------- |
| `job_id` | string   | 是           | 任务唯一标识，全局唯一，用于后续状态查询 |

---

## 🔍 2. 查询任务状态接口

用于轮询某个任务的实时状态、loss 数据及运行耗时。前端可定时调用该接口更新任务进度。

### 接口信息

| **项目**     | **说明**                      |
| ------------ | ----------------------------- |
| **接口路径** | `/api/v1/train_jobs/{job_id}` |
| **请求方法** | `GET`                         |
| **鉴权方式** | Bearer Token                  |

### 请求参数

| **参数位置** | **参数名** | **类型** | **必填** | **说明**                |
| ------------ | ---------- | -------- | -------- | ----------------------- |
| Path         | `job_id`   | string   | 是       | 创建任务时返回的任务 ID |

### 响应示例

#### 任务进行中

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "job_id": "j_20260424_abc123",
    "is_completed" : true,
    "status": "completed",
    "loss_data": {
      ###自由定义只需要将所有的loss放置于这个属性中
    },
    "duration": 1280
  }
}
```

#### 任务已完成

任务完成后，`duration` 表示该任务最终花费的总耗时（秒），不再随时间变化。

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "job_id": "j_20260424_abc123",
    "is_completed" : true,
    "status": "completed",
    "loss_data": {
      ###自由定义只需要将所有的loss放置于这个属性中
    },
    "duration": 1864
  }
}
```

### 响应字段说明

| **字段**                 | **类型**      | **是否必返** | **说明**                                                     |
| ------------------------ | ------------- | ------------ | ------------------------------------------------------------ |
| `job_id`                 | string        | 是           | 当前任务 ID                                                  |
| `is_completed`           | boolean       | 是           | 任务是否完成                                                 |
| `status`                 | string (enum) | 是           | 当前任务状态，取值见下方枚举表                               |
| `loss_data`              | object        | 是           | 当前任务的 loss 数据集合，统一封装                           |
| `loss_data.steps`        | int[]         | 是           | 训练步数序列，与 `values` 一一对应                           |
| `loss_data.values`       | float[]       | 是           | 每个步数对应的 loss 值                                       |
| `loss_data.current_loss` | float         | 是           | 最新一次 loss 值，便于前端快速展示                           |
| `duration`               | int           | 是           | 任务耗时（单位：秒）。运行中时为**已持续时间**；已完成时为**最终总耗时** |

### `status` 枚举值

`status` 字段为字符串枚举类型，建议覆盖完整生命周期：

| **枚举值**  | **含义** | **说明**                                   |
| ----------- | -------- | ------------------------------------------ |
| `pending`   | 等待中   | 任务已创建，尚未开始执行                   |
| `running`   | 执行中   | 任务正在运行，loss 持续更新                |
| `completed` | 已完成   | 任务正常结束，`duration` 为最终耗时        |
| `failed`    | 执行失败 | 任务异常终止，建议配合错误详情字段返回     |
| `cancelled` | 已取消   | 用户主动取消或系统中断                     |
| `earlyout`  | 提前终止 | 用户提前终止模型（暂时没有实现这方面功能） |

## ⚠️ 错误码规范

建议对两个接口统一使用如下业务错误码，便于前端分支处理。

| **code** | **HTTP 状态** | **message**       | **说明**            |
| -------- | ------------- | ----------------- | ------------------- |
| `0`      | 200           | success           | 请求成功            |
| `40001`  | 400           | invalid parameter | 参数缺失或格式错误  |
| `40101`  | 401           | unauthorized      | 未登录或 Token 失效 |
| `40401`  | 404           | job not found     | `job_id` 不存在     |
| `50001`  | 500           | internal error    | 服务端内部错误      |

---

## 💡 设计要点小结

几个容易被忽略但很关键的约定，建议落地时重点把关：

- **`duration` 字段语义双关**：运行中表示「已持续时间」，完成后表示「最终耗时」，前端可直接根据 `status` 判断展示文案（如「已运行 xx 秒」/「总耗时 xx 秒」）。
- **`loss_data` 统一封装**：未来如需扩展（如 accuracy、learning_rate），可平滑加入同一对象，避免字段散落。
- **`job_id` 建议带业务前缀 + 时间戳 + 随机串**：如 `j_20260424_abc123`，便于追溯与排查。

这份规范可直接作为团队内部接口契约文档，也方便后续扩展到任务日志、任务取消等衍生接口 ✅

## 📦 3. 模型发布接口

用于将已完成训练任务的模型目录发布到统一存储位置，并返回模型文件相对路径。

### 接口信息

| **项目**        | **说明**             |
| --------------- | -------------------- |
| **接口路径**    | `/api/model/publish` |
| **请求方法**    | `POST`               |
| **鉴权方式**    | Bearer Token         |
| **Content-Type** | `application/json`  |

### 请求参数

#### Headers

| **参数名**      | **类型** | **必填** | **说明**                               |
| --------------- | -------- | -------- | -------------------------------------- |
| `Authorization` | string   | 是       | 鉴权 Token，格式：`Bearer {token}`     |
| `Content-Type`  | string   | 是       | 固定值：`application/json`             |

#### Body

```json
{
  "user_id": 10001,
  "version": "1.0.0",
  "job_id": "train_job_20260121103000"
}
```

| **参数名** | **类型** | **必填** | **示例**                       | **说明**                                     |
| ---------- | -------- | -------- | ------------------------------ | -------------------------------------------- |
| `user_id`  | number   | 是       | `10001`                        | 用户唯一标识 ID                              |
| `version`  | string   | 是       | `"1.0.0"`                      | 模型版本号，语义化版本格式（主.次.修订）     |
| `job_id`   | string   | 是       | `"train_job_20260121103000"`   | 训练或构建任务 ID                            |

### 成功响应

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "model_path": "models/user_10001/v1.0.0/train_job_20260121103000/model.bin"
  }
}
```

| **参数名**   | **类型** | **说明**                               |
| ------------ | -------- | -------------------------------------- |
| `model_path` | string   | 模型文件相对路径，可用于后续下载或加载 |

### 失败响应示例

#### 参数校验失败

```json
{
  "code": 500,
  "message": "invalid version format, expected x.y.z",
  "data": null
}
```

#### 权限不足

```json
{
  "code": 401,
  "message": "no permission to publish model",
  "data": null
}
```

#### 任务不存在

```json
{
  "code": 404,
  "message": "job_id not found",
  "data": null
}
```

### 错误码说明

| **错误码** | **含义**                |
| ---------- | ----------------------- |
| `200`      | 成功                    |
| `401`      | 请求参数非法 / 未授权   |
| `404`      | 任务不存在              |
| `500`      | 服务端内部错误          |

### 备注

- `model_path` 为相对路径，根目录由服务端配置决定。
- 同一 `user_id + version + job_id` 多次调用，返回路径保持一致（覆盖发布）。

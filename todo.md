# 模型推理接口改造 TODO（实施版）

## 0. 约束与边界

1. 接口兼容约束
- 现有训练、发布、查询接口路径、请求参数、响应主结构不变。
- 新增推理接口必须单独成路由，不破坏旧接口语义。
- `/api/*` 兼容接口仍需遵循《接口文档规范说明.md》：
  - 外层结构保持 `code/message/data`
  - 字段命名保持 `snake_case`
  - 业务错误码与错误消息可扩展，但不能改旧字段含义

2. 模型来源约束
- 推理必须基于已训练、已发布或已落盘的模型目录。
- 不允许直接依赖训练中的临时产物目录。
- 推理层应支持按 `model_path` 直连，预期是直接使用发布的模型。
- 发布接口返回值应为发布目录的绝对路径，便于推理接口直接定位模型。

3. 输入输出约束
- 前端传入的是“训练数据之外的新数据”，接口必须对输入做严格校验。
- 首版优先支持单次推理，后续再扩展批量推理。
- 返回值必须可直接给前端渲染，不能只返回内部对象。

## 1. 接口契约设计

1. 新增推理接口
- 新增一个独立接口，例如：
  - `POST /api/model/infer`
  - 或 `POST /api/model/predict`
- 接口名称、请求结构、响应结构先定稿，再开始实现。

2. 请求参数设计
- 必须明确以下信息：
  - `model_path`：发布后的模型绝对路径
  - `cov_group`：要预测的相关组，其中每个元素为`{"target":"...","covariates":["..."]}`
  - `prediction_length`：预测步数
  - `csv_path`：推理所需的新数据 CSV 文件路径

3. 响应结构设计
- 必须包含：
  - `code`
  - `message`
  - `data`
- `data` 建议至少包含：
  - `predictions`：预测结果组成的列表，每个元素的格式为`{"target":"...","prediction":[0.9, 0.8, ...]}`
  - 多个 `target` 时按 `cov_group` 顺序逐个返回

## 3. 推理服务层改造

1. 新增推理服务模块
- 新建 `app/services/inference_service.py`。
- 统一封装：
  - 模型加载
  - 输入校验
  - 推理执行
  - 输出格式化

2. 复用训练侧能力
- 复用现有 `model_service.load_local_model()` 或抽出更通用的加载逻辑。
- 复用数据预处理思路，但不要直接耦合训练输入格式。

3. 输入预处理
- 从 `csv_path` 读取前端输入，转换成推理所需的时间序列格式。
- 校验：
  - 历史长度是否足够
  - target 是否存在
  - 协变量列是否齐全
  - 时间顺序是否正确
- 对异常输入给出明确错误信息。

4. 推理执行
- 统一封装一次推理流程：
  - 加载模型
  - 组织输入
  - 调用预测
  - 解析输出
  - 返回前端可用结果
- 每个 `cov_group` 需要根据 `target` 选择微调任务中对应的模型。
- 一次推理包含多个 `target` 时，按组逐个调用对应模型，再汇总结果返回。

## 4. API 与鉴权

1. 路由接入
- 新增 `app/api/inference.py`。
- 在 `app/main.py` 注册新路由。

2. 鉴权接入
- 继续沿用现有 `Bearer Token` 鉴权。
- 与现有 `/api/model/publish` 保持一致。

3. 异常处理
- 推理接口必须统一返回规范响应。
- 处理以下错误：
  - 模型不存在
  - 参数非法
  - 输入数据不完整
  - 推理执行失败
  - 模型加载失败

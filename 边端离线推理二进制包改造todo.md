# 边端离线推理二进制包改造 TODO

## 1. 背景

当前系统已经具备模型训练、模型发布和前端调用推理接口的能力。  
在在线服务部署场景下，前端可以通过 HTTP 接口调用 `/api/model/infer` 完成推理。

但是在实际使用中，可能存在边端设备或离线部署场景，例如：

- 设备无法稳定联网；
- 客户现场不希望部署完整后端服务；
- 只需要在本地读取模型和 CSV 文件完成预测；
- 希望通过命令行脚本集成到其他系统、批处理任务或工业现场流程中；
- 希望交付一个不要求用户手动配置 Python 环境和依赖的可执行程序。

因此，需要将现有推理能力从 Web API 中抽离出来，形成一个可以独立运行的命令行推理程序，并进一步支持打包为离线可执行包。

---

## 2. 核心目标

本次改造的核心目标是：

> 将推理逻辑封装为可独立运行的 CLI 程序，并支持后续打包为边端离线可执行包。

该可执行包只包含：

- 推理逻辑；
- 必要运行时依赖；
- 模型加载逻辑；
- metadata 读取逻辑；
- CSV 读取和校验逻辑；
- 推理结果导出逻辑。

该可执行包不包含：

- 训练数据；
- 推理 CSV；
- 已发布模型权重；
- 用户业务数据；
- 固定模型路径。

模型路径、输入 CSV 路径、输出路径等均由命令行参数传入。

---

## 3. 推荐使用方式

改造后，边端设备上应能够通过类似以下命令完成推理：

```bash
chronos_infer \
  --model-path ./models/train_job_001 \
  --csv-path ./data/input.csv \
  --output-path ./result/output.json
```

其中：

- `--model-path` 指向已发布模型目录；
- `--csv-path` 指向待推理的输入 CSV；
- `--output-path` 指向推理结果输出路径。

如果模型目录中已经存在 `metadata.json`，程序应自动读取其中的：

- `selected_groups`
- `target`
- `covariates`
- `prediction_length`
- `context_length`
- 子模型目录信息

从而避免用户在边端再次手动输入这些配置。

---

## 4. 总体设计思路

### 4.1 推理逻辑应从 FastAPI 路由中抽离

当前推理逻辑如果直接写在 API 路由中，将不利于复用。  
本次改造应将核心推理逻辑抽象成一个可被多种入口调用的公共服务。

推荐目标结构是：

```text
HTTP 推理接口
    ↓
公共推理服务
    ↓
模型加载 / 数据读取 / 滚动窗口推理 / 结果生成

CLI 推理入口
    ↓
公共推理服务
    ↓
模型加载 / 数据读取 / 滚动窗口推理 / 结果生成
```

也就是说：

- Web API 和 CLI 不应该各自维护一套推理逻辑；
- 二者应该调用同一个推理服务；
- API 负责处理 HTTP 请求和响应；
- CLI 负责处理命令行参数和文件输出；
- 核心推理服务只关注推理本身。

---

### 4.2 二进制包只封装逻辑，不封装模型和数据

边端部署时，推荐交付形态为：

```text
edge_package/
├── bin/
│   └── chronos_infer
├── models/
│   └── train_job_001/
│       ├── metadata.json
│       ├── finetuned-ckpt_value1/
│       └── finetuned-ckpt_value2/
├── data/
│   └── input.csv
└── result/
```

运行时通过参数指定模型和数据：

```bash
./bin/chronos_infer \
  --model-path ./models/train_job_001 \
  --csv-path ./data/input.csv \
  --output-path ./result/output.json
```

这样可以实现：

- 程序和模型解耦；
- 换模型不需要重新打包程序；
- 换输入数据不需要修改程序；
- 多个模型可以共用同一个推理程序；
- 更适合离线部署和版本管理。

---

### 4.3 CLI 结果格式应尽量与 HTTP 接口保持一致

为了减少前后端、边端和集成系统之间的适配成本，CLI 输出结果建议尽量复用现有 `/api/model/infer` 的响应结构。

例如输出 JSON：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "model_path": "./models/train_job_001",
    "csv_path": "./data/input.csv",
    "predictions": [
      {
        "target": "value1",
        "prediction": [0.1, 0.2, 0.3],
        "actual": [10.1, 10.4, 10.9]
      }
    ]
  }
}
```

失败时也应输出结构化错误，例如：

```json
{
  "code": 400,
  "message": "history length is insufficient",
  "data": null
}
```

CLI 同时应通过退出码表达成功或失败：

- 成功：退出码 `0`
- 参数错误、文件不存在、推理失败等：非 `0`

具体错误码设计由实现者根据现有项目风格决定。

## 5. CLI 参数设计建议

不要把具体实现写死，但建议至少支持以下参数。

### 5.1 必需参数

| 参数 | 说明 |
|---|---|
| `--model-path` | 已发布模型目录 |
| `--csv-path` | 推理输入 CSV 文件路径 |
| `--output-path` | 推理结果输出路径 |

### 5.2 可选参数

| 参数 | 说明 |
|---|---|
| `--device` | 推理设备，例如 `cpu` 或 `cuda` |
| `--format` | 输出格式，例如 `json`，后续可扩展为 `csv` |
| `--context-length` | 覆盖模型 metadata 中的默认 `context_length` |
| `--prediction-length` | 覆盖模型 metadata 中的默认 `prediction_length` |
| `--targets` | 只推理部分 target |
| `--verbose` | 输出更详细日志 |
| `--version` | 显示 CLI 程序版本 |

其中：

- 默认情况下应优先从 `metadata.json` 读取推理配置；
- 用户通过命令行显式传入的参数可以覆盖 metadata；
- 如果 metadata 不存在，则应允许用户通过命令行提供必要参数；
- 如果 metadata 和命令行参数都不足以完成推理，应给出清晰错误。

### 6.2 新增 CLI 推理入口

- [ ] 新增一个命令行推理入口。
- [ ] CLI 入口负责解析命令行参数。
- [ ] CLI 入口负责调用公共推理服务。
- [ ] CLI 入口负责将结果写入 `--output-path`。
- [ ] CLI 入口负责打印必要的运行信息。
- [ ] CLI 入口应在成功时返回退出码 `0`。
- [ ] CLI 入口应在失败时返回非 `0` 退出码。
- [ ] CLI 入口应尽量输出结构化错误，方便脚本或其他系统集成。

---

### 6.3 复用 metadata 自动配置能力

- [ ] CLI 推理应支持读取 `<model-path>/metadata.json`。
- [ ] 如果 metadata 中存在 `selected_groups`，默认使用它作为推理分组。
- [ ] 如果 metadata 中存在 `prediction_length`，默认使用它作为预测长度。
- [ ] 如果 metadata 中存在 `context_length`，默认使用它作为上下文长度。
- [ ] 如果命令行显式传入覆盖参数，则优先使用命令行参数。
- [ ] 如果 metadata 缺失但命令行参数完整，仍应支持推理。
- [ ] 如果 metadata 和命令行参数都不足，应返回清晰错误。

---

### 6.4 文件和路径校验

- [ ] 校验 `--model-path` 是否存在。
- [ ] 校验 `--csv-path` 是否存在。
- [ ] 校验 `--output-path` 的父目录是否存在或可创建。
- [ ] 校验模型目录中是否存在对应 target 的子模型目录。
- [ ] 校验输入 CSV 中是否包含所需 target 列。
- [ ] 校验输入 CSV 中是否包含所需 covariate 列。
- [ ] 校验相关列是否为数值列。
- [ ] 校验输入数据长度是否满足 `context_length` 要求。
- [ ] 错误信息应面向边端用户，避免只输出 Python traceback。

---

### 6.5 输出结果

- [ ] 默认输出 JSON 文件。
- [ ] JSON 输出结构尽量与 `/api/model/infer` 响应一致。
- [ ] 输出中应包含推理结果列表。
- [ ] 输出中应包含每个 target 的预测结果。
- [ ] 如果当前 HTTP 接口返回 `actual`，CLI 输出也应保持一致。
- [ ] 可以预留扩展点，后续支持 CSV 输出。
- [ ] 写入输出文件前，应确保目录存在或自动创建目录。
- [ ] 输出完成后，可在命令行打印输出文件路径。

---

### 6.6 日志与错误处理

- [ ] CLI 应提供基本日志输出。
- [ ] 普通模式下日志不要过于冗长。
- [ ] `--verbose` 模式下可以输出更详细的调试信息。
- [ ] 失败时应输出清晰错误原因。
- [ ] 对常见错误进行友好提示，例如：
  - [ ] 模型路径不存在；
  - [ ] CSV 路径不存在；
  - [ ] metadata 缺失；
  - [ ] CSV 缺列；
  - [ ] 子模型目录不存在；
  - [ ] 输入数据长度不足；
  - [ ] 模型加载失败；
  - [ ] 推理执行失败。
- [ ] CLI 不应在常见错误下直接暴露大量内部堆栈信息。
- [ ] 如需调试堆栈，可仅在 `--verbose` 模式下输出。

---

### 6.7 打包支持

本任务不要求一次性完成所有平台的打包，但需要为后续二进制包交付预留条件。

- [ ] 确保 CLI 可以通过 Python 命令直接运行。
- [ ] 确保 CLI 入口可以被 PyInstaller / Nuitka 等工具识别。
- [ ] 不要在 CLI 中写死开发机路径。
- [ ] 不要依赖当前工作目录作为唯一运行方式。
- [ ] 所有模型和数据路径都应来自命令行参数。
- [ ] 避免在推理逻辑中隐式访问训练服务专用目录。
- [ ] 如果有资源文件依赖，应明确集中管理。

建议优先支持 `onedir` 风格的可执行包，而不是强制单文件：

```text
dist/
└── chronos_infer/
    ├── chronos_infer
    ├── runtime dependencies
    └── other required libraries
```

原因：

- PyTorch / Chronos / Transformers 依赖较重；
- 单文件模式可能启动慢；
- 动态库路径问题更难排查；
- `onedir` 更适合离线交付和问题定位。

## 7. 不建议做的事情

本次改造不建议：

- [ ] 不要把模型权重打进二进制包。
- [ ] 不要把 CSV 数据打进二进制包。
- [ ] 不要为每个模型单独打一个二进制程序。
- [ ] 不要在 CLI 中重新实现一套和 HTTP 接口不同的推理逻辑。
- [ ] 不要让 CLI 强依赖 FastAPI 服务启动。
- [ ] 不要要求边端设备必须联网下载依赖。
- [ ] 不要把开发环境中的绝对路径写死到代码中。
- [ ] 不要让用户必须手动安装完整 Python 环境后才能运行最终交付物。
